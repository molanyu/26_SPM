# Scenario Testing Spec

## 本轮验收阻塞修正规则

以下规则用于修复手动验收中发现的自动流程歧义：

- 未签到后的系统行为统一表述为“超时未签到释放”，不是“学生取消预约”，也不应在学生端继续显示“取消预约”。
- 流程测试和手动验收应验证后台调度器自动推进提醒、超时未签到释放、违约落账和通知落账；不得把管理端手动触发按钮作为业务流程的一部分。
- 若需要加速时间，自动化测试应使用 injected now、fake clock 或 scheduler `tick/run_once`；真实手动联调则必须启用后台调度器并等待真实时间。
- 场景测试必须增加同一学生/用户同一时间段预约两个不同座位被拒绝的验收项。

## 1. 任务定位

本文档定义本项目在整体验收阶段需要补充的跨模块业务流程测试边界。

前期模块测试主要用于验证各模块是否满足自身接口与业务规则，例如 `reservation` 负责预约规则，`checkin` 负责签到规则，`notification` 负责提醒规则。

流程测试不替代模块测试。流程测试用于验证多个模块在同一条业务时间线中协作时，学生端、管理端、通知、违约和统计结果是否保持一致。

## 2. 实现范围

本阶段流程测试只覆盖课程作业中最核心的自习室预约时间线。

本阶段纳入范围：

- 学生登录
- 学生查询自习室与座位
- 学生创建预约
- 预约开始前提醒
- 学生签到
- 未签到提醒
- 超时未签到释放
- 违约记录生成
- 座位释放
- 管理端预约记录查询
- 管理端违约记录查询
- 管理端当前动态签到码状态与签到记录核对
- 管理端通知日志核对
- 管理端使用统计查询

本阶段不纳入范围：

- 真实浏览器 UI 自动化
- 微信开发者工具或真实小程序自动化
- 真实 SMTP 邮件发送
- 真实等待时间流逝
- 性能压测
- 多日复杂排班或调度优化

## 3. 测试规则

流程测试必须保持可重复、可定位、可自动执行。

测试规则固定如下：

- 不允许依赖真实时间等待。
- 必须使用固定时间或已有服务中的 `now` 参数模拟时间推进。
- 默认通知通道必须使用 `mock` 或 fake sender。
- 真实 SMTP 只用于显式 smoke/manual 验证，不进入默认流程测试。
- 测试可以通过公开 HTTP API 和公开 service 入口组织流程。
- 测试不得绕过业务 service 直接制造最终状态。
- 允许在测试开始前通过 fixture 准备基础数据，例如院系、学生、自习室、座位和系统参数。

## 4. 首批流程场景

### 4.1 SCN-01 正常履约流程

目标：

验证学生能够完成一次正常预约，并且提醒、签到、管理端记录和统计结果能够保持一致。

流程：

1. 准备一个启用的院系、学生、自习室和座位。
2. 学生登录。
3. 学生查询目标时间段内可用自习室和座位。
4. 学生创建符合 30 分钟粒度且开始时间在未来的预约。
5. 在预约开始前的提醒时间点运行预约提醒任务。
6. 学生在允许签到时间窗口内完成签到。
7. 管理员查询当前动态签到码状态和签到记录。
8. 管理员查询预约记录。
9. 管理员查询使用统计。
10. 管理员查询通知日志，确认提醒任务已经落账。

验收结果：

- 预约创建成功。
- 预约提醒只生成一次。
- 签到成功。
- 管理端可查看指定自习室当前 5 分钟动态签到码状态和该预约签到记录。
- 管理端预约记录中可以查询到该预约。
- 使用统计中体现该预约对应的使用时长。
- 重复运行提醒任务不会重复生成成功通知。

### 4.2 SCN-02 未签到违约与座位释放流程

目标：

验证学生未按时签到时，系统能够完成提醒、超时未签到释放、违约记录、座位释放和管理端查询闭环。

流程：

1. 准备一个启用的院系、学生、自习室和座位。
2. 学生登录。
3. 学生创建符合 30 分钟粒度且开始时间在未来的预约。
4. 在未签到提醒时间点运行未签到提醒任务。
5. 在超时未签到释放时间点运行 scheduler `tick/run_once`。
6. 管理员查询通知日志，确认未签到提醒和超时未签到释放通知已经落账或能说明失败原因。
7. 管理员查询预约记录。
8. 管理员查询违约记录。
9. 学生或管理员查询该座位后续时间段状态。

验收结果：

- 未签到提醒只生成一次。
- 预约按超时未签到规则置为 `EXPIRED` 并释放座位。
- 违约记录只生成一次。
- 座位在后续兼容时间段内重新可用。
- 管理端预约记录中可以看到取消状态。
- 管理端违约记录中可以在无筛选时看到全部违约，也可以按学生学号、自习室和日期任意组合筛选对应违约。

## 5. 手动验收阻塞修复场景

本阶段流程测试和手动验收必须覆盖以下回归边界：

- `FAIL-01`：预约时间粒度为 30 分钟，`09:30-10:30` 可创建，`09:15-10:15` 必须被拒绝。
- `FAIL-02`：学生创建预约和管理员代预约都必须拒绝过去或已经开始的时间段。
- `FAIL-03`：违约管理页默认显示全部违约记录，`user_id`、`student_no`、`room_id`、`date_from`、`date_to` 均为独立可选筛选条件，HTML 页面错误不得裸露 JSON。
- `FAIL-04`：管理端必须提供最小签到入口，支持查看指定自习室当前 5 分钟动态签到码状态、有效至时间和签到记录；不得提供生成签到码、补签或代签到动作。
- `FAIL-05`：学生端助手必须展示面向用户的卡片、列表、空状态和错误状态，不得把后端结构化结果直接渲染为 raw JSON。
- `FAIL-06`：通知由后台调度器或内部 `tick/run_once` 推进；管理端必须提供可观察的通知日志，不提供手动触发任务入口。

## 6. 文件范围

本阶段建议新增以下测试目录：

```text
tests/scenario/
```

建议首批测试文件：

```text
tests/scenario/test_reservation_lifecycle.py
```

后续如需纳入统一测试入口，可在稳定后扩展：

```text
python run_tests.py --suite scenario
```

是否将 `scenario` 纳入默认 `all` 测试集，应在流程测试稳定、执行时间可控后再决定。

## 7. 完成标准

流程测试第一版完成时，必须满足：

- `tests/scenario/` 中至少覆盖 `SCN-01` 和 `SCN-02`。
- 测试不依赖真实时间等待。
- 测试不依赖真实 SMTP、Docker 专属状态或人工交互。
- 测试能够同时验证学生端可观察结果和管理端可观察结果。
- 任一关键模块交接失败时，流程测试必须失败。
- 流程测试失败时，应能从断言信息中定位是哪一段业务时间线断裂。
