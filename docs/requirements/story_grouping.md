# 用户故事分组

## 1. 文档定位

本文件用于将现有用户故事重组为可交付单元。

分组目标如下：

- 让每组故事在业务上高内聚
- 降低组与组之间的直接耦合
- 让每组都可以形成相对完整的开发与验收闭环
- 为后续模块开发顺序、任务拆分和验收安排提供依据

## 2. 分组原则

- 优先按业务闭环分组，不按页面分组
- 优先把同一核心状态流转中的故事放在一起
- 基础能力先交付，上层能力后交付
- 查询类能力尽量依附于已有业务模块，不单独拆出过小交付单元

## 3. 交付单元

### G1. 身份与权限

目标：

- 建立用户身份、院系归属、角色权限和管理端菜单控制基础能力

包含故事：

- US-09
- US-17
- US-18
- US-19

对应模块：

- `identity`

交付结果：

- 用户身份识别
- 院系访问限制基础能力
- 角色与权限管理
- 管理端按权限展示菜单

依赖关系：

- 作为基础单元，被其他业务单元依赖

### G2. 资源管理与可见性

目标：

- 建立自习室、座位、开放时间和座位属性的基础资源模型

包含故事：

- US-01
- US-02
- US-07
- US-11
- US-12

对应模块：

- `resource`

交付结果：

- 学生可查看自习室和座位
- 管理员可维护自习室和座位
- 支持靠窗、插座等属性筛选

依赖关系：

- 依赖 G1
- 被预约、签到、智能查询等单元依赖

### G3. 系统规则配置

目标：

- 抽离系统级业务参数，避免规则散落在各模块中

包含故事：

- US-13

对应模块：

- `system_config`

交付结果：

- 最大预约时长
- 签到宽限时间
- 违约判定参数

依赖关系：

- 依赖 G1
- 被预约、签到、违约、提醒单元依赖

### G4. 预约核心链路

目标：

- 建立预约、取消、代预约、代取消和历史复用的核心业务闭环

包含故事：

- US-03
- US-08
- US-10
- US-15

对应模块：

- `reservation`

交付结果：

- 学生创建预约
- 学生取消预约
- 管理员代预约与代取消
- 历史预约查询与再次预约

依赖关系：

- 依赖 G1、G2、G3
- 是后续签到、违约、提醒的前置单元

### G5. 签到与超时释放

目标：

- 建立动态码签到和预约状态变更能力

包含故事：

- US-04
- US-06
- US-23

对应模块：

- `checkin`
- `violation`

交付结果：

- 动态码或二维码签到
- 超时未签到自动取消
- 自动生成违约记录

依赖关系：

- 依赖 G4
- 与 G6 配合完成提醒与记录查询

### G6. 提醒与记录查询

目标：

- 建立预约提醒、违约记录查询和运营查询基础能力

包含故事：

- US-05
- US-14

对应模块：

- `notification`
- `violation`

交付结果：

- 预约前提醒
- 未签到提醒
- 管理员查询预约记录和违约记录

依赖关系：

- 依赖 G4、G5

### G7. 统计分析

目标：

- 基于已有记录构建使用率和违约率统计能力

包含故事：

- US-16

对应模块：

- `violation`
- `resource`

交付结果：

- 自习室使用率统计
- 座位使用率统计
- 违约率统计

依赖关系：

- 依赖 G6

### G8. 智能查询

目标：

- 建立自然语言查询入口，将查询请求转发为内部业务查询

包含故事：

- US-20
- US-21
- US-22

对应模块：

- `assistant`

交付结果：

- 查询今晚空座
- 查询靠窗座位
- 查询本人今日预约

依赖关系：

- 依赖 G2、G4

### G9. 自动化测试与流水线

目标：

- 建立自动化测试和 CI/CD 基础能力

包含故事：

- US-24

对应模块：

- `tests`
- `devops`

交付结果：

- 单元测试
- 集成测试
- 流水线接入

依赖关系：

- 贯穿全部单元
- 每个交付单元完成后都必须补对应测试

## 4. 交付顺序

固定顺序如下：

1. G1 身份与权限
2. G2 资源管理与可见性
3. G3 系统规则配置
4. G4 预约核心链路
5. G5 签到与超时释放
6. G6 提醒与记录查询
7. G7 统计分析
8. G8 智能查询
9. G9 自动化测试与流水线

说明：

- G1、G2、G3 是基础层
- G4 是核心业务单元
- G5、G6 建立预约后的完整闭环
- G7、G8 属于增强能力
- G9 不单独等待最后一次性完成，而是贯穿全程持续补齐

## 5. 最小验收闭环

每个交付单元完成时，至少要具备：

- 对应数据模型
- 对应 service
- 对应接口或页面
- 至少一组可执行测试
- 对应用户故事验收标准

没有形成最小闭环的故事组，不进入下一个交付单元。
